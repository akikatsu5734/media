## 1. 結論

### 一言で言うと
**理想フローの70〜80%は、Claude Code + 既存外部ツールで実現可能。ただし「完全自動化」より「人間承認を1箇所挟む半自動」が、品質・リスク・コストの観点から現実的かつ推奨。**

### 何が実現できるか
| 実現レベル | 内容 |
|-----------|------|
| **今すぐ試せる** | 話題収集・タイトル案生成・記事下書き生成・WordPress下書き投稿 |
| **1〜2ヶ月で実現可能** | 分析→改善提案ループ・地域情報の自動拡充・スケジュール投稿 |
| **3〜6ヶ月で実現可能** | 公開後分析の自動化・次テーマ提案の精度向上・問診フロー連携 |
| **現時点では難しい** | 完全無人での公開・リアルタイム一次情報確認・高品質な画像自動生成 |

### 人間承認を必ず残すべき箇所（3箇所）
1. **テーマ選定・タイトル選択**: AIが候補を出し、人間が選ぶ。AIだけで決めると方向性のズレが蓄積する。
2. **記事公開前の最終確認**: 事実確認・トーン確認・法的表現の確認は人間が行う。
3. **改善施策の承認**: 分析→改善提案はAIが出すが、実施するかどうかは人間が判断する。

### 2週間で試せること
話題収集 → タイトル案生成 → 記事下書き → WordPress下書き投稿 の一連フローをPoC実施できる。  
必要なもの: WordPress APIキー・Claude APIキー・RSSフィードのURLリスト（3〜5本）。

## 2. 全体フローの分解

理想フローの10工程を、4段階で評価する。

| # | 工程 | 評価 | 理由 |
|---|------|------|------|
| 1 | **話題収集** | Claude Code + 外部ツール | RSS・公式APIで自動収集可能。スクレイピングは規約リスクがあるため使用しない |
| 2 | **テーマ選定** | Claude Codeで候補生成 + 人間確認が必須 | AIが3〜5候補を提案し、最終判断は人間が行う。AIだけで決めると質のばらつきが出る |
| 3 | **タイトル案生成** | Claude Code で可能 | キーワード・季節性・地域性を考慮した複数案を自動生成できる |
| 4 | **人間承認** | 人間確認が必須 | タイトル選択・記事方向性の最終決定は人間が行う。この箇所は自動化しない |
| 5 | **記事生成** | Claude Code で可能 | 構成案→本文→メタ情報まで一貫して生成できる。ただし事実確認は人間が担う |
| 6 | **WordPress連携** | Claude Code + REST API | WordPress REST APIで下書き投稿・タグ・カテゴリ設定まで自動化できる |
| 7 | **スケジュール投稿** | Claude Code + WordPress API で可能 | `status: future` + `date` パラメータで実現可能。ただし公開前確認ステップを推奨 |
| 8 | **公開後分析** | Claude Code + GA4/Search Console API | 両APIで記事ごとの指標を自動取得・集計できる |
| 9 | **次テーマ提案** | Claude Code で可能 | 分析結果を入力に「次に書くべき記事候補」をAIが生成できる |
| 10 | **運用改善提案** | Claude Code で可能（人間確認推奨） | 改善提案はAIが生成可能だが、実施可否は人間が判断する体制を維持する |

### 自動化できない理由が明確な箇所
- **一次情報確認**: ハルシネーション（AIの事実誤認）は現時点では完全には防げない。法律・制度・補助金情報は必ず人間が一次ソースと照合する。
- **画像生成**: 品質・著作権・ブランド整合性の問題から、アイキャッチ画像の完全自動生成は対象外とする。
- **完全無人公開**: 記事の品質・倫理・SEO的適切さを担保するために、最終公開の承認は人間が行う。

## 3. 話題収集の実現方法

### 基本方針
**スクレイピングは原則使用しない。** 利用規約違反・サイト構造変更によるメンテナンスコスト・法的リスクを避けるため、公式RSS・公式APIのみを使用する。

### 収集ソース一覧

| ソース | 種別 | 内容 | コスト |
|--------|------|------|--------|
| NHK ニュース RSS | RSS | 空き家・不動産・行政関連ニュース | 無料 |
| 国土交通省 新着情報 RSS | RSS | 空き家対策・住宅政策・補助金情報 | 無料 |
| 不動産ジャパン ニュース | RSS | 不動産市況・地価・空き家関連 | 無料 |
| Google Alerts (RSS出力) | RSS | 「空き家」「空き家活用」等キーワードでカスタム収集 | 無料 |
| e-Stat API（総務省） | 公式API | 住宅・土地統計調査、空き家数データ | 無料・利用規約クリア |
| 国土交通省 土地情報ライブラリ | 公開データ | 地価・不動産取引情報 | 無料 |
| 各区公式広報 RSS | RSS | 23区それぞれの行政トピック・補助金情報 | 無料 |

### 23区地域トピックの収集方法

```text
収集フロー:
1. 各区の公式広報ページのRSSを登録（23件）
2. Claude Codeがキーワードフィルタリング（「空き家」「住宅」「補助金」等）
3. 地域別に整理してレポート出力
4. 週1回、担当者に通知
```

仮説: 23区すべてのRSS登録は初期設定に1〜2時間かかるが、その後は完全自動。

### 季節性テーマの生成方法

毎月の定番テーマをClaude Codeに事前生成させ、「季節性テーマカレンダー」として管理する。

```text
例（月別テーマ候補）:
1月: 「相続・年明けに考える空き家整理」「固定資産税の納付と空き家の関係」
3月: 「年度末に動く不動産市況と空き家売却タイミング」
7月: 「夏の空き家管理・草刈り・害虫対策」
10月: 「空き家活用の秋・補助金申請シーズン」
12月: 「年末に考える空き家の行く末・相続前に確認すべきこと」
```

このカレンダーをベースに、収集した時事ニュースと掛け合わせてテーマを生成する。

### 収集の自動化設計（最小MVP）

```text
[定期実行: 週1回]
  ↓
RSS取得スクリプト（Claude Code）
  ↓
キーワードフィルタリング（空き家・不動産・補助金・相続 等）
  ↓
重複排除・優先度スコアリング
  ↓
「今週の話題候補レポート」（Markdown）を出力
  ↓
担当者がSlackまたはメールで確認
```

定期実行は `cron` または GitHub Actions（無料枠）で対応可能。

## 4. タイトル案提案フロー

### AIによるテーマ判断の基準

Claude Codeが「今出すべきテーマ」を判断する際に使う基準：

| 基準 | 内容 |
|------|------|
| **季節性・タイムリー性** | 今の時期に検索されやすいか（例: 確定申告シーズンの固定資産税） |
| **地域性** | 特定の区・地域に関連する行政トピックや補助金情報があるか |
| **ユーザー課題との整合** | 空き家所有者が「今困っていること」に直接答えるテーマか |
| **競合記事の薄さ** | 既存のWebコンテンツが少なく、上位表示の余地があるか（仮説ベース） |
| **サイト内の未カバー領域** | 既存記事リストと照合して、まだ書いていないテーマか |
| **過去の高読了記事との類似性** | GA4で読まれている記事のテーマに近いか |

### タイトル案の生成ルール

1テーマにつき、以下の3軸で各1〜2案を生成（計3〜5案）：

| 軸 | 例 |
|----|---|
| **SEO重視** | 「世田谷区 空き家 補助金 2026年最新版」 |
| **ユーザー課題重視** | 「相続した実家を放置すると固定資産税が6倍になる理由」 |
| **地域特化** | 「世田谷区の空き家バンク登録方法と活用事例まとめ」 |

### 人間が受け取るレポートの形式

```markdown
## 今週のテーマ候補（2026-04-24）

### テーマ候補 1
- **推奨タイトル案A（SEO）**: 「空き家 固定資産税 特例 2026年最新情報」
- **推奨タイトル案B（課題）**: 「空き家を持ち続けると税金がどう変わるのか」
- **推奨タイトル案C（地域）**: 「世田谷区の空き家と固定資産税軽減措置の現状」
- **選定理由**: 2026年1月に固定資産税特例の見直し報道あり（ソース: NHK RSS）
- **想定キーワード**: 空き家 固定資産税 特例 2026
- **競合状況**: 関連記事は少なく上位表示の余地あり（仮説）
- **既存記事との重複**: なし

→ 選んでください: A / B / C / 不採用
```

### 承認フローの設計（最小MVP）

```text
Claude Code → Markdownレポート生成
  ↓
メール or Slack通知（添付またはリンク）
  ↓
担当者がレポートを見て番号を返信
  ↓
Claude Codeが選ばれたタイトルで記事生成を開始
```

最初はメールやSlackへの自動通知は不要。レポートをファイルとして保存し、担当者が確認→次のコマンドを実行するだけでも機能する。

## 5. 記事生成〜WordPress投稿フロー

### 記事生成の工程

```text
入力: 選ばれたテーマ・タイトル・想定キーワード
  ↓
STEP 1: 構成案の生成（見出しH2・H3の一覧）
  ↓
STEP 2: 本文生成（各セクションを順次生成）
  ↓
STEP 3: メタ情報生成
  - メタディスクリプション（120字以内）
  - OGP用サマリー
  - カテゴリ候補（既存カテゴリから選択）
  - タグ候補（5〜10個）
  - アイキャッチ画像の指示文（Canva等での制作用）
  ↓
STEP 4: 品質チェック（Claude Codeによる自己レビュー）
  - 事実確認が必要な箇所にフラグを立てる
  - 法的表現（「必ず」「保証」等）の過剰使用を検出
  - CTA（行動喚起）の有無を確認
  ↓
STEP 5: WordPressへ下書き投稿
```

### WordPress REST API による投稿

```python
# 下書き投稿の例（Python）
import requests
import base64

wp_url = "https://aki-katsu.co.jp/magazine/wp-json/wp/v2/posts"
credentials = base64.b64encode(b"username:app_password").decode("utf-8")

headers = {
    "Authorization": f"Basic {credentials}",
    "Content-Type": "application/json"
}

post_data = {
    "title": "選ばれたタイトル",
    "content": "生成された本文（HTML形式）",
    "status": "draft",          # 下書き投稿
    "categories": [カテゴリID],
    "tags": [タグID1, タグID2],
    "excerpt": "メタディスクリプション"
}

response = requests.post(wp_url, json=post_data, headers=headers)
```

認証は **WordPress Application Password**（管理画面から発行可能・無料）を使用。外部プラグイン不要。

### スケジュール投稿の実現方法

```python
# スケジュール投稿（例: 翌朝10時公開）
from datetime import datetime, timedelta

publish_time = (datetime.utcnow() + timedelta(days=1)).replace(hour=1, minute=0)  # UTC+9→UTC変換

post_data = {
    "status": "future",
    "date": publish_time.isoformat()  # 例: "2026-04-25T01:00:00"
}
```

WordPressの `status: future` + `date` パラメータで実現可能。追加プラグイン不要。

### アイキャッチ画像の扱い

**自動生成は現時点では対象外。** 理由：

- AI画像生成（Midjourney等）は品質にばらつきがあり、ブランドトーン管理が難しい
- 著作権・利用規約の確認コストが発生する
- Canvaでのテンプレート制作（人間）のほうが品質・コストともに現実的

**Claude Codeが生成するのは「画像制作指示文」のみ：**

```text
アイキャッチ指示文の例:
- メインビジュアル: 古い一軒家の外観（日本の住宅街・夕方）
- テキストオーバーレイ: 「固定資産税と空き家」
- カラーイメージ: 落ち着いた青・白・グレー系
- Canvaテンプレート: ブログ記事アイキャッチ（1200×630px）
```

### 人間確認ポイントのまとめ

```text
[自動] 話題収集 → テーマ候補生成
[人間] ← テーマ・タイトル選択（5〜10分）
[自動] 記事生成 → 下書き投稿
[人間] ← 記事の事実確認・文章確認・アイキャッチ作成（20〜30分）
[人間] ← 公開ボタンを押す（または公開日時を設定）
```

従来の記事制作（ゼロから書く: 3〜5時間）が、**確認・編集作業（30〜60分）に圧縮**される。

## 6. 公開後分析フロー

### 取得できるデータと用途

| データソース | 取得できる指標 | 活用方法 |
|------------|--------------|---------|
| **GA4 API** | セッション数・直帰率・平均滞在時間・ページビュー | 記事ごとの読了状況を把握 |
| **Search Console API** | 表示回数・クリック数・CTR・平均掲載順位・検索クエリ | 流入キーワード・改善余地の把握 |
| **両者の組み合わせ** | 「流入は多いが離脱が多い記事」「流入は少ないが滞在が長い記事」等の分類 | 優先改善対象の特定 |

### 自動分析スクリプトの出力イメージ

```markdown
## 週次記事パフォーマンスレポート（2026-04-24）

### 📈 今週のハイライト
- 最多流入記事: 「世田谷区 空き家 補助金」（セッション: 312）
- 最高CTR記事: 「固定資産税 空き家 特例」（CTR: 8.2%）
- 改善推奨記事: 「空き家活用 賃貸」（表示回数: 890、CTR: 0.8% → タイトル・メタ改善余地あり）

### 🔍 次に書くべき記事候補
1. 「空き家バンク 登録方法 世田谷」（検索クエリで流入あり、対応記事なし）
2. 「空き家 売却 税金 2026」（表示回数増加中、上位表示の余地あり）
3. 「相続 空き家 手続き 費用」（関連クエリが季節的に増加）

### ⚠️ アクション推奨
- TASK: 「空き家活用 賃貸」のタイトルとメタディスクリプションを改善
- TASK: 「空き家バンク 登録方法 世田谷」の新規記事を作成
```

### 改善ループの設計

```text
[週1回]
GA4 / Search Console API からデータ取得
  ↓
記事ごとの分類
- 流入多い × 反応良い → 類似テーマを増やす
- 流入多い × 反応悪い → タイトル・構成改善
- 流入少ない × 反応良い → 内部リンク強化
- 流入少ない × 反応悪い → 優先度を下げる
  ↓
Claude Code が改善提案レポートを生成
  ↓
人間が実施可否を判断
  ↓
次週の記事生成テーマへ反映
```

## 8. リスクと対策

| # | リスク | 深刻度 | 対策 |
|---|--------|--------|------|
| 1 | **ハルシネーション（事実誤認）** | 高 | 法律・制度・数値の記述には「要確認」フラグを自動付与。公開前に人間が一次ソースを確認する |
| 2 | **一次情報確認不足** | 高 | 補助金・行政制度の情報は必ず自治体公式サイトのURLを引用元として明記する |
| 3 | **ニュース選定ミス** | 中 | AIが収集した話題は「候補」であり、最終選定は人間が行う。センシティブな話題（事故・犯罪）は自動除外ルールを設ける |
| 4 | **地域性の薄い記事量産** | 中 | テーマ生成時に「地域性スコア」を付与し、地域情報を含まない記事は優先度を下げる |
| 5 | **WordPress投稿事故** | 中 | 初期は必ず `status: draft`（下書き）のみで投稿。`status: publish` の直接投稿は当面禁止 |
| 6 | **SEO品質のブレ** | 中 | 記事生成プロンプトにSEO必須要素（キーワード密度・見出し構造・内部リンク）を組み込む。定期的にプロンプトを見直す |
| 7 | **Claude APIコスト増** | 低〜中 | 月次でAPIコストをモニタリング。記事1本あたりのコスト（目安: 30〜100円）を把握して予算管理する |
| 8 | **運用の属人化** | 中 | スクリプト・プロンプト・手順書を `docs/` と `prompts/` で管理。誰でも再現できる状態を維持する |
| 9 | **スクレイピングによる規約違反** | 高（該当する場合） | スクレイピングは使用しない。すべてRSS・公式APIに限定する |
| 10 | **生成コンテンツのGoogleペナルティ** | 中 | AI生成のみで公開しない。必ず人間が編集・加筆する「AI支援コンテンツ」として品質を担保する |

## 9. 2週間で試すべきPoC案

### PoCの目的
「話題収集 → タイトル案生成 → 記事下書き → WordPress下書き投稿」の一連フローが、**実際に動くこと**を確認する。

### 実施内容

| Day | やること | 成果物 |
|-----|---------|--------|
| Day 1〜2 | WordPressのApplication Passwordを発行し、REST APIへの接続テストを実施 | APIからの下書き投稿成功 |
| Day 3〜4 | Google Alerts（「空き家」「空き家 補助金」）のRSSを取得するスクリプトを実装 | 最新10件の話題が取得できるスクリプト |
| Day 5〜7 | 収集した話題をもとにClaude Codeがタイトル案3〜5本を生成するスクリプトを実装 | タイトル候補レポート（Markdown） |
| Day 8〜10 | 選ばれたタイトルで記事本文・メタ情報を生成するスクリプトを実装 | 記事下書き（Markdown） |
| Day 11〜12 | 生成した記事をWordPressに下書き投稿するスクリプトを実装・テスト | WordPress上に下書き記事が存在する |
| Day 13〜14 | 一連フローを通しで実行し、品質・コスト・工数を計測 | PoCレポート（所感・数値・改善点） |

### 成功条件
- [ ] 一連フローが1回エラーなく通しで動作する
- [ ] 記事下書きの品質が「インターンが書いたドラフト以上」と判断される
- [ ] 記事1本あたりのAIコストが200円以下に収まる
- [ ] 担当者の確認作業が30分以内に完了できる

### 失敗条件（撤退・方針変更の判断基準）
- 記事品質が低すぎて確認・修正に1時間以上かかる
- APIコストが想定の3倍以上になる
- WordPressへの投稿でデータ破損・文字化けが発生する

### 意思決定ポイント（Day 14）
PoC完了後に以下を判断する：

1. **そのままフェーズ2へ進む**: 品質・コスト・工数がすべて成功条件を満たした場合
2. **プロンプトを改善して再試行**: 品質に問題があった場合（1〜2週間で対応可能）
3. **アプローチを変更する**: コストまたは工数が想定を大きく超えた場合

## 10. 湯浅さん向け要約

**アキカツの記事制作を、AIでどこまで自動化できるか**

現在、記事1本を書くのに3〜5時間かかっていますが、Claude Codeを使った半自動ワークフローを構築すれば、**担当者の作業を「確認・承認の30〜60分」に圧縮できます**。

具体的には、AIが毎週ニュースや行政情報を自動収集し、「今書くべきテーマ」の候補を提案します。担当者はその中から選ぶだけ。選ばれたテーマをもとにAIが記事を下書き生成し、WordPressに自動で投稿します。担当者は内容を確認して公開ボタンを押すだけです。

**完全な自動化は目指しません。** 事実確認と最終承認は必ず人間が行います。AIはあくまで「速く・安く・大量に下書きを作る」役割です。これにより、記事の制作コストを下げながら、地域情報や時事テーマへの対応スピードを上げることができます。

まず**2週間の小さな実験**から始め、うまくいけば段階的に自動化の範囲を広げます。初期コストはClaude APIの利用料（月3,000〜5,000円程度）のみです。大きな開発投資は不要で、今すぐ試せます。
